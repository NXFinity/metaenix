Error Handling

Understanding how Meta EN|IX API handles and returns errors is crucial for building robust integrations.

Error Response Format​

All errors follow a consistent structure:

{
  "statusCode": 400,
  "message": [
    "Error message here"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/users/me"
}

Error Response Fields​

FieldTypeDescription
statusCodenumberHTTP status code
messagestring[]Array of error messages
timestampstringISO 8601 timestamp when error occurred
pathstringAPI endpoint where error occurred
errorstringError name (development only)
stackstringStack trace (development only)

HTTP Status Codes​

400 Bad Request​

The request was malformed or invalid.

Common Causes:

  • Missing required fields
  • Invalid data format
  • Validation errors

Example:

{
  "statusCode": 400,
  "message": [
    "username must be a string",
    "email must be an email"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/auth/register"
}

Handling:

try {
  const response = await fetch('https://api.metaenix.com/v1/auth/register', {
    method: 'POST',
    body: JSON.stringify(data)
  });
  
  if (!response.ok) {
    const error = await response.json();
    if (response.status === 400) {
      // Handle validation errors
      error.message.forEach(msg => console.error(msg));
    }
  }
} catch (error) {
  console.error('Request failed:', error);
}

401 Unauthorized​

Authentication required or invalid credentials.

Common Causes:

  • Missing access token
  • Invalid or expired token
  • Invalid credentials

Example:

{
  "statusCode": 401,
  "message": [
    "Unauthorized"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/users/me"
}

Handling:

if (response.status === 401) {
  // Token expired or invalid
  // Refresh token or redirect to login
  await refreshAccessToken();
  // Retry request
}

403 Forbidden​

Authenticated but not authorized for this resource.

Common Causes:

  • Insufficient permissions
  • Resource access denied
  • Scope mismatch

Example:

{
  "statusCode": 403,
  "message": [
    "You don't have permission to access this resource"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/users/123/posts"
}

404 Not Found​

Resource not found.

Common Causes:

  • Invalid resource ID
  • Resource deleted
  • Incorrect endpoint

Example:

{
  "statusCode": 404,
  "message": [
    "User not found"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/users/invalid-id"
}

429 Too Many Requests​

Rate limit exceeded.

Example:

{
  "statusCode": 429,
  "message": [
    "Too many requests. Limit: 5 per 900 seconds"
  ],
  "limit": 5,
  "remaining": 0,
  "resetAt": "2025-01-15T10:15:00.000Z",
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/auth/login"
}

Handling:

if (response.status === 429) {
  const error = await response.json();
  const resetTime = new Date(error.resetAt);
  const waitTime = resetTime.getTime() - Date.now();
  
  // Wait before retrying
  await new Promise(resolve => setTimeout(resolve, waitTime));
  // Retry request
}

500 Internal Server Error​

Server-side error occurred.

Example:

{
  "statusCode": 500,
  "message": [
    "Internal server error"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/posts"
}

Handling:

if (response.status === 500) {
  // Log error for monitoring
  logError(error);
  // Implement retry with exponential backoff
  await retryWithBackoff(request);
}

Validation Errors​

Validation errors return multiple messages:

{
  "statusCode": 400,
  "message": [
    "username must be a string",
    "username must be longer than or equal to 3 characters",
    "email must be an email",
    "password must be longer than or equal to 8 characters"
  ],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/auth/register"
}

Handling Validation Errors:

function handleValidationErrors(error) {
  if (error.statusCode === 400 && Array.isArray(error.message)) {
    error.message.forEach(msg => {
      // Display each validation error to user
      console.error(`Validation error: ${msg}`);
    });
  }
}

Error Handling Best Practices​

1. Always Check Response Status​

const response = await fetch(url, options);

if (!response.ok) {
  const error = await response.json();
  // Handle error
}

2. Implement Retry Logic​

async function fetchWithRetry(url, options, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await fetch(url, options);
      
      if (response.ok) {
        return await response.json();
      }
      
      // Don't retry client errors (4xx)
      if (response.status >= 400 && response.status < 500) {
        throw new Error(`Client error: ${response.status}`);
      }
      
      // Retry server errors (5xx)
      if (i < maxRetries - 1) {
        await sleep(Math.pow(2, i) * 1000); // Exponential backoff
        continue;
      }
      
      throw new Error(`Server error: ${response.status}`);
    } catch (error) {
      if (i === maxRetries - 1) throw error;
      await sleep(Math.pow(2, i) * 1000);
    }
  }
}

3. Handle Rate Limits​

async function handleRateLimit(response) {
  if (response.status === 429) {
    const error = await response.json();
    const resetTime = new Date(error.resetAt);
    const waitTime = Math.max(0, resetTime.getTime() - Date.now());
    
    console.log(`Rate limited. Waiting ${waitTime}ms until ${resetTime}`);
    await new Promise(resolve => setTimeout(resolve, waitTime));
    return true; // Indicates retry should happen
  }
  return false;
}

4. Log Errors Appropriately​

function logError(error, context) {
  // Log to monitoring service
  if (error.statusCode >= 500) {
    // Server errors - log for investigation
    monitoringService.logError(error, context);
  } else if (error.statusCode === 429) {
    // Rate limits - log for optimization
    monitoringService.logRateLimit(error, context);
  }
  // Client errors (4xx) - usually don't need logging
}

5. User-Friendly Error Messages​

function getUserFriendlyMessage(error) {
  const statusMessages = {
    400: 'Please check your input and try again',
    401: 'Please log in to continue',
    403: 'You don\'t have permission to perform this action',
    404: 'The requested resource was not found',
    429: 'Too many requests. Please wait a moment and try again',
    500: 'Something went wrong. Please try again later'
  };
  
  return statusMessages[error.statusCode] || 'An error occurred';
}

Common Error Scenarios​

Authentication Errors​

// Token expired
if (response.status === 401) {
  try {
    await refreshToken();
    // Retry original request
  } catch (refreshError) {
    // Redirect to login
    redirectToLogin();
  }
}

Network Errors​

try {
  const response = await fetch(url);
} catch (error) {
  if (error.name === 'TypeError' && error.message.includes('fetch')) {
    // Network error
    console.error('Network error - check connection');
  }
}

Timeout Errors​

const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 5000);

try {
  const response = await fetch(url, {
    signal: controller.signal
  });
} catch (error) {
  if (error.name === 'AbortError') {
    console.error('Request timeout');
  }
} finally {
  clearTimeout(timeoutId);
}

Development vs Production​

In development, error responses include additional debugging information:

{
  "statusCode": 500,
  "message": ["Internal server error"],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/posts",
  "error": "InternalServerErrorException",
  "stack": "Error: ..."
}

In production, sensitive information is hidden:

{
  "statusCode": 500,
  "message": ["Internal server error"],
  "timestamp": "2025-01-15T10:00:00.000Z",
  "path": "/v1/posts"
}
